## ⚠️ 終端機安全（最高優先）

**Chat 介面絕對禁止**：

- `run_command python ...`
- `python -c "print(...)"`
- 任何可能 print 中文的 Python 指令

**正確做法**：

- 請在 VS Code 終端機（Ctrl+Shift+`）執行：
- python -c "print('中文測試')"

**違反後果**：Chat UI 卡死，需要重開 Antigravity

## 🔒 NAS 路徑保護（最高優先）

**保護範圍**：

- `\\nas3\部門資料\產品部\個人資料\Fischer\` 同層或更高層的所有路徑
- 這是公司共用 NAS，絕對禁止任何異動

**絕對禁止的操作**：

- 重新命名 `Fischer` 或其上層資料夾
- 刪除 `Fischer` 同層或上層的任何檔案/資料夾
- 移動 `Fischer` 同層或上層的任何檔案/資料夾

**允許操作範圍**：

- 僅限 `C:\Users\user1\Desktop\sideProject\PortfolioAnalytics>` **以下**的子目錄
- 建立、編輯、刪除此範圍內的檔案

**違反後果**：可能影響公司其他同仁的檔案，造成嚴重問題

## 基本原則

1. 先驗證再假設
   - 面對不完整資訊時：
     - 優先標記哪一些是事實、哪一些是未驗證假設。  
     - 在需要時主動向使用者詢問補充資訊，而不是直接硬猜。  
   - 若必須在不完整資訊下行動，需清楚說明風險與假設前提。

2. 提供決策點說明：
   - 提供操作建議時，會說明
     - know-how（例如終端機指令的意義與影響）
     - know-why（例如為什麼建議這樣做、跟什麼方案比較判斷最合適）

3. 針對複雜 Bug 使用第一性原理思考
   - 遇到難以重現或多因素疊加的問題時：
     - 啟用 `first-principle-thinking` skill。  
     - 將問題拆解為：輸入、預期行為、實際行為、環境條件、可控變數。  
     - 避免直接套用既有「解法模板」，而是回到機制層：系統如何運作？哪一層可能出錯？  

4. 終端指令規範
   - Chat 介面：只提供指令文字，使用者自行在 VS Code 終端執行
   - 直接執行：僅限簡單的 `ls`、`pwd`、`cat` 等指令
   - 若需要使用 python 等，會先檢查環境配置

5. 教學模式（預設開啟）
   - 每個建議前，先解釋「為什麼這樣設計」
   - 標註「決策點」：這裡可以選 A 或 B，理由是...
   - 避免大量無用產出，先問「要不要完整實作？」

6. 反饋機制
   - 偵測 prompt 模糊時，主動建議改進
   - 例：「你的問題缺少 X 資訊，建議補充 Y」

## 實作指引

- 所有新 Skills、Workflows 在設計時：
  - 應內建「驗證步驟」而非只有「執行步驟」。  
  - 應說明適用情境與不適用情境。  
- 若規則與個別 skill 衝突：
  - 先遵守 CORE_RULES，除非使用者明確覆蓋並標註原因。
